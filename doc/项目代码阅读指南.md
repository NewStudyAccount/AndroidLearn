# AndroidLearn 项目代码阅读指南

> 本指南帮助你高效地阅读和理解本项目中的教学代码。
> 它不是知识教程（教程请看 [四大组件学习教程.md](./四大组件学习教程.md)），
> 而是**"先读哪个文件、每个文件看什么、代码与概念如何对应"**的导航手册。

---

## 一、项目定位

本项目是一个 **Android 四大组件综合演示**，用最小可运行代码分别演示：

- **Activity** —— 生命周期、页面跳转、数据传递、返回结果
- **Service** —— 启动式服务、绑定式服务
- **BroadcastReceiver** —— 系统广播、自定义广播、动态注册
- **ContentProvider** —— 自定义 Provider + SQLite 数据源

所有演示入口集中在 `MainActivity`，通过按钮逐个触发，便于边看代码边运行观察。

## 四、逐文件阅读指引

### 4.1 `AndroidManifest.xml` —— 组件注册清单

**看什么**：四大组件必须在清单文件中声明才能生效，这是 Android 的硬性规则。

**重点关注**：
| 节点 | 关键属性 | 含义 |
|------|----------|------|
| `<activity>` | `android:exported` | 是否允许其他应用启动 |
| `<service>` | `android:exported="false"` | 仅应用内部可启动 |
| `<receiver>` | `<intent-filter>` | 静态注册监听 `BOOT_COMPLETED` |
| `<provider>` | `android:authorities` | 唯一标识 URI 主机名 |

**思考题**：为什么 `MainActivity` 的 `exported=true`？为什么其他 Activity 是 `false`？

---

### 4.2 `MainActivity.kt` —— 演示总控台

**看什么**：这是整个项目的"遥控器"，每个按钮触发一个组件演示。

**阅读建议**：先快速浏览 `setupUI()` 方法，建立"按钮 → 组件"的映射表：

| 按钮 | 触发的组件 | 对应方法 |
|------|-----------|----------|
| `btnLifecycle` / `btnDataPass` | Activity | `startActivity(intent)` |
| `btnStartService` / `btnStopService` | Service(start) | `startService()` / `stopService()` |
| `btnBindService` / `btnUnbindService` | Service(bind) | `bindService()` / `unbindService()` |
| `btnRegisterReceiver` / `btnSendBroadcast` | Receiver | `registerReceiver()` / `sendBroadcast()` |
| `btnQueryProvider` | Provider | `contentResolver.query()` |

**重点学习 `onDestroy()`**：它演示了组件生命周期的"清理职责"——解绑 Service、注销 Receiver，防止内存泄漏。

---

### 4.3 `activity/ActivityLifecycleDemo.kt` —— 生命周期

**看什么**：每个生命周期方法都打了一条 `Log.d` 日志。

**动手实践**：
1. 运行 App，点击"生命周期演示"按钮
2. 打开 Logcat，过滤标签 `LifecycleDemo`
3. 分别操作：进入 → 按 Home → 返回 → 按返回键
4. 对照日志与代码，理解回调顺序

**配套阅读**：[四大组件学习教程.md](./四大组件学习教程.md) 第 1.1 节生命周期图。

---

### 4.4 `activity/DataPassDemo.kt` —— 数据传递四种方式

**看什么**：四种数据传递方式集中在一个文件，便于横向对比。

| 方式 | 关键 API | 特点 |
|------|----------|------|
| 简单数据 | `intent.putExtra(key, value)` | 直接存基本类型 |
| Bundle | `Bundle().putXxx()` → `intent.putExtras()` | 批量打包 |
| 对象 | `intent.putExtra("user", user)` | 需 `@Parcelize` |
| 返回结果 | `registerForActivityResult` + `launcher.launch()` | 异步回调 |

**重点理解 `User` 数据类**：使用 `@Parcelize` 注解自动实现 `Parcelable` 接口，这是 Android 传递对象的标准方式（比 `Serializable` 高效）。

**配套阅读**：[Kotlin语法详解.md](./Kotlin语法详解.md) 中的 data class 与注解章节。

---

### 4.5 `activity/ReceiveDataActivity.kt` —— 接收数据

**看什么**：与 `DataPassDemo` 一一对应，演示如何从 `intent` 中取回数据。

**注意细节**：
- `intent.getStringExtra("name")` 返回可空类型，用 `if (name != null)` 判空
- 取对象用 `getParcelableExtra`，注意 API 33+ 的兼容写法（本项目已修复）

---

### 4.6 `activity/PickUserActivity.kt` —— 返回结果

**看什么**：演示"请求-响应"模式的接收方。

**核心三步**：
```kotlin
setResult(RESULT_OK, resultIntent)   // 1. 设置结果码 + 数据
finish()                              // 2. 关闭自己
// 3. 数据回到 DataPassDemo 的 pickUserLauncher 回调中
```

---

### 4.7 `service/MyService.kt` —— 启动式服务

**看什么**：`startService` 启动的服务独立运行，与调用者解耦。

**重点关注 `onStartCommand()`**：
- 启动一个子线程模拟后台任务（注意：Service 默认运行在主线程！）
- 任务完成后调用 `stopSelf()` 自行停止
- 返回值 `START_STICKY` 决定被杀后的重启策略

**思考题**：为什么不能直接在 `onStartCommand` 里写耗时逻辑？答案在代码注释里。

---

### 4.8 `service/MyBindService.kt` —— 绑定式服务

**看什么**：`bindService` 启动的服务与调用者绑定，可双向通信。

**核心机制**：通过 `LocalBinder` 内部类返回 Service 实例，让 Activity 能直接调用 Service 的方法：
```kotlin
inner class LocalBinder : Binder() {
    fun getService(): MyBindService = this@MyBindService
}
```

**配套阅读**：回到 `MainActivity.kt` 的 `serviceConnection` 对象，看 `onServiceConnected` 如何拿到 `MyBindService` 实例并调用 `getData()`、`getCount()`。

---

### 4.9 `receiver/MyReceiver.kt` —— 广播接收器

**看什么**：`onReceive` 用 `when` 表达式按 action 分发处理。

**注意约束**：
- `onReceive` 运行在主线程，**不能**执行耗时操作（>10s 会 ANR）
- `context` 参数可空，调用前需判空

**两种注册方式对照**：
| 方式 | 位置 | 代码 |
|------|------|------|
| 静态注册 | AndroidManifest.xml | `<receiver>` + `<intent-filter>` |
| 动态注册 | MainActivity.kt | `registerReceiver(myReceiver, filter)` |

---

### 4.10 `database/UserDatabaseHelper.kt` —— SQLite 数据源

**看什么**：这是 `ContentProvider` 的底层数据来源。

**核心方法**：
- `onCreate(db)` —— 首次创建时执行建表 SQL + 插入示例数据
- `onUpgrade(db, old, new)` —— 版本升级时回调（本项目简单删表重建）

**理解后再读 Provider**：ContentProvider 只是包装层，真正存数据的是 SQLite。

---

### 4.11 `provider/UserContentProvider.kt` —— 内容提供者

**看什么**：本项目最复杂的文件，建议最后阅读。

**核心知识点**：

1. **URI 匹配机制**：
   ```
   content://com.example.androidlearn.provider/users       → 匹配码 100（全部）
   content://com.example.androidlearn.provider/users/5     → 匹配码 101（单个）
   ```
   通过 `UriMatcher` 把 URI 字符串映射成整数分支。

2. **六个必须重写的方法**：`onCreate` / `query` / `insert` / `update` / `delete` / `getType`，分别对应 CRUD + 初始化 + MIME 类型。

3. **数据变化通知**：`context?.contentResolver?.notifyChange(uri, null)` 通知观察者数据已更新。

**配套阅读**：[四大组件学习教程.md](./四大组件学习教程.md) 第 4 章 ContentProvider。

## 六、动手练习建议

阅读只是第一步，建议按以下顺序动手实践以加深理解：

### 练习 1：扩展 Activity（入门）
在 `DataPassDemo` 中新增一个按钮，传递一个 `List<String>` 给 `ReceiveDataActivity` 并显示。

### 练习 2：扩展 Service（进阶）
给 `MyBindService` 添加一个 `startTimer()` 方法，每秒回调一次计数到 Activity，并在界面显示。

### 练习 3：扩展广播（进阶）
新增一个自定义广播 action，让 `MyReceiver` 收到后通过 `LocalBroadcastManager`（或显式 Intent）回传给 MainActivity 更新 UI。

### 练习 4：扩展 Provider（高级）
在 `UserContentProvider` 中新增一个 `searchByName` 的 URI 路径（如 `users/name/xiaoming`），支持按姓名模糊查询。

## 七、运行与调试

### 运行项目
1. 用 Android Studio 打开项目根目录
2. 等待 Gradle Sync 完成
3. 连接真机或启动模拟器（API ≥ 24）
4. 点击 Run 按钮

### 查看 Service / Receiver 日志
打开 Logcat，分别过滤以下标签：

| 标签 | 来源 |
|------|------|
| `MainActivity` | 主界面交互日志 |
| `LifecycleDemo` | Activity 生命周期日志 |
| `MyService` | 启动式服务日志 |
| `MyBindService` | 绑定式服务日志 |
| `MyReceiver` | 广播接收日志 |
| `UserContentProvider` | 内容提供者日志 |
| `UserDBHelper` | 数据库操作日志 |

### 验证 ContentProvider
点击"查询 ContentProvider"按钮，状态区会显示 SQLite 中的示例用户列表。若为空，检查 `UserDatabaseHelper.onCreate` 是否执行（首次创建数据库时调用）。

## 八、常见问题排查

| 现象 | 可能原因 | 排查方法 |
|------|----------|----------|
| 点击按钮无反应 | 监听器未绑定 | 检查 `findViewById` 的 ID 是否匹配布局 |
| 跳转后崩溃 | 目标 Activity 未在 Manifest 注册 | 检查 `AndroidManifest.xml` |
| Service 启动崩溃 | 未在 Manifest 注册 | 检查 `<service>` 节点 |
| 广播收不到 | 未注册或 action 不匹配 | 比对 `addAction` 与发送方 action |
| 注册广播崩溃 | Android 13+ 需指定标志 | 已在本项目修复，参考 `MainActivity.kt` |
| Provider 查询为空 | 数据库未初始化 | 卸载重装 App 触发 `onCreate` |
